# 動画→棋譜化サービス: 認識API側への要求仕様（ドラフト）

対局動画（または連続写真）から**手順付きの棋譜（消費時間込みKIF）**を復元するサービスを
計画しています。フロント側の設計と、API側にお願いしたい機能を優先度順にまとめます。

## 前提: フロント側の役割分担（動画はアップロードしない）

対局動画は95%以上の時間で盤面が静止しているため、動画そのものは送りません。
ブラウザ内で以下を行い、**キーフレーム（安定局面の静止画）だけ**を送ります。

1. `<video>`+canvas で低解像度2fpsサンプリング → フレーム間差分で動きスコア算出
2. 手が盤上にある間はスコアが高い＝遮蔽フレームは自然に除外
3. 1.5秒以上静止＆前回採用フレームと十分差がある → 高解像度JPEGを1枚採用
4. 結果: **100手の対局 ≈ キーフレーム100〜150枚（10〜20MB）+ タイムスタンプ列**

これにより通信量は動画比1/100以下、API側は「画像バッチ処理」に還元されます。
盤面以外の映像（人物・会場）が端末外に出ない点は、行橋の撮影ガイドラインとも整合します。

## 要求一覧（優先度順）

### R1. バッチ認識エンドポイント【必須・フェーズ1】

```
POST /recognize/batch
Body: multipart (frames.zip) または {s3_presigned_url}
Params: 既存と同じ (model / waku / mochi_crop / mochi_ocr / hidden_rotate)
Response: { results: [ { frame_index, ban_result, sente_mochi, gote_mochi, teban }, ... ] }
```

- 単発 `/recognize` をフロントから150回逐次実行するのは、コールドスタート・
  レート制限・API Gateway 30秒制限の面で非現実的
- GPU的にも1セッションにまとめた方が総コストが下がるはず

### R2. 非同期ジョブAPI【必須・フェーズ1】

```
POST /jobs            → { job_id }
GET  /jobs/{job_id}   → { status: queued|processing|done|failed, progress: 0-100, result_url }
```

- 同期レスポンスは30秒制限があるため不可能
- 結果は7日程度保持（フロントはjob_idをlocalStorageに持ち「閉じても後で確認できる」UXにする）
- 可能であればwebhook通知（無くても可）

### R3. マスごとの確信度【強く希望・フェーズ1】

```
"ban_confidence": { "76": 0.98, "22": 0.61, ... },
"mochi_confidence": { "sente": {"fu": 0.9}, ... }
```

- 「どの手をユーザーに確認させるか」「時系列ソルバがどこを疑うか」の燃料
- 500枚診断を実施できたということは、内部的に確度情報をお持ちのはず

### R4. 枠ヒントの持ち越し【希望・フェーズ1】

```
Param: waku_hint = [[x1,y1],[x2,y2],[x3,y3],[x4,y4]]  (盤の4隅・1フレーム目の検出結果)
```

- 固定カメラでは盤は動かないため、2フレーム目以降の枠検出を省略/初期化でき、
  精度と速度が同時に上がる

### R5. 時系列整合ソルバ【フェーズ2・まずはフロント実装で検証】

局面列＋時刻列 → 合法手列の復元。

- 駒数保存則を**時間方向に**適用（`joint` の時系列版）
- キーフレーム間で2手進んでいた場合は合法手探索でブリッジ
- 曖昧箇所は勝手に選ばず `{ ply, alternatives: [...] }` で返す

フェーズ0〜1ではフロント側JSで自前実装して精度を検証し、
アルゴリズムが固まった段階でAPI側への移管を相談させてください。
（駒箱誤認識のような系統誤差は、時系列では「最初から一貫して存在する駒」として
むしろ除去しやすくなるはず — ここは一緒に設計したいポイントです）

### R6. コスト・スループットの見積り【相談】

- 1フレームあたりの処理単価（こちらの料金設計の前提になります）
- バッチの最大枚数、150枚の想定所要時間
- バッチのコールドスタート挙動

### R7. データ保持ポリシーの明文化【B2B展開の前提】

- アップロードされたフレームの保存期間・利用目的
- 大会運営（B2B）に提案する際に必ず聞かれる項目です

## フェーズ計画（フロント側）

| フェーズ | 内容 | API側への要求 |
|---|---|---|
| 0 | 連続写真モード（1手ごとに撮影→既存APIを逐次呼び→JS手順復元） | **なし（既存APIのまま）** |
| 1 | 動画→キーフレーム抽出＋バッチ処理 | R1, R2（+R3, R4） |
| 2 | サーバサイド時系列ソルバ＋B2B機能 | R5 |

フェーズ0は既にプロトタイプがあります（`public/renzoku.html`）。
手順復元ロジック（合法手マッチング・駒数保存則トラッキング）はここで検証中です。
