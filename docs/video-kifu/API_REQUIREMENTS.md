# 動画→棋譜化サービス: 認識API側への要求（改訂版）

対局動画から**手順付きの棋譜（消費時間込みKIF）**を復元するサービスを計画しています。

**前提（改訂）**: 動画の受け取り・キーフレーム抽出・手順復元・ジョブ管理は
**こちらのAWSで自前運用**します（設計は `PIPELINE.md` 参照）。
認識は既存の単発 `POST /recognize` をサーバ側から呼び続ける形にするため、
**当初検討していたバッチAPI・非同期ジョブAPIの新設は不要になりました。**

その上で、お願いしたいのは以下の4点です。

## 1. マスごとの確信度【最重要】

```
Response追加案:
"ban_confidence":   { "76": 0.98, "22": 0.61, ... },
"mochi_confidence": { "sente": {"fu": 0.9}, ... }
```

- 用途: 補正エディタで「どの手をユーザーに確認させるか」のハイライト、
  手順復元ソルバの許容誤差の重み付け
- 100枚を超える局面列を扱うと、閾値なしの全確認はUX的に破綻するため、
  これが精度体感を最も左右します
- 500枚診断を実施できたということは、内部的には確度情報をお持ちのはず

## 2. レート・同時実行数の合意

- サーバ側ループから `/recognize` を呼びます（1ジョブ=100〜150リクエスト）
- こちらの並列数上限・リトライ間隔をそちらの都合に合わせるので、
  「同時Nまで・秒間Mまで」の目安をください
- あわせて1リクエストあたりの単価目安も（料金設計の前提にします）

## 3. 枠ヒントの持ち越し（waku_hint）【あると精度・速度が上がる】

```
Param追加案: waku_hint = [[x1,y1],[x2,y2],[x3,y3],[x4,y4]]
```

- 固定カメラの動画では盤が動かないため、1フレーム目の枠検出結果を
  2フレーム目以降に引き継げると、枠ズレ起因の誤認識が減りGPU時間も節約できるはず

## 4. 学習データ利用の相談

- 動画由来のキーフレーム（実対局・多様な盤駒・照明）は学習データとして
  価値があるはずです
- こちら側でユーザーから**明示同意を取った分だけ**を提供する設計にしたいので、
  欲しいデータの形式・アノテーションの粒度があれば教えてください
- 動画そのものは当面こちらのS3で保持します（ライフサイクル・削除ポリシーは
  `PIPELINE.md` 参照）

## 補足: フェーズ0の実績

連続写真→手順復元のプロトタイプ（`public/renzoku.html`）で、
合法手マッチング・読み飛ばしブリッジ・駒数保存則トラッキングは検証済みです。
`joint=0` で呼んでいます（時系列整合はこちら側で取るため、単発の保存則補正と
二重にならないようにする意図）。この分担で問題ないかもご意見ください。
