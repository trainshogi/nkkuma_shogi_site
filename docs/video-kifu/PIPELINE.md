# 動画→棋譜化: 自前AWSパイプライン設計

動画のアップロードとキーフレーム抽出・手順復元は**こちらのAWSで完結**させ、
認識APIは既存の単発 `POST /recognize` をそのまま使う。
（API提供側への要求が「バッチAPI・ジョブAPIを作って」から「確信度を返して」程度まで軽くなる）

## 全体図

```
ブラウザ                         こちらのAWS                            API提供側(既存)
────────                        ─────────────                          ──────────
動画を選ぶ
 └ POST /uploads ──────────────▶ API GW+Lambda: presigned URL発行
 └ S3マルチパート直接アップロード ▶ S3 videos/{job}/raw.mp4
                                  └ S3イベント → Step Functions
                                     ① 抽出 Lambda (ffmpegレイヤ)
                                        2fps縮小サンプリング+フレーム差分
                                        → 安定局面キーフレーム 100〜150枚
                                        → S3 frames/{job}/NNN.jpg
                                     ② 認識 Lambda
                                        キーフレームを並列数制限つきで ────▶ POST /recognize
                                        503リトライ・レート制御を一元化 ◀──── (単発・既存のまま)
                                     ③ 手順復元 Lambda (Node)
                                        renzoku.js の stitch()/buildKif() を再利用
                                        → S3 results/{job}/kifu.json
GET /jobs/{id} ◀───────────────── API GW + DynamoDB(ジョブ状態)
補正エディタで修正 → KIF完成
(renzoku.html の結果画面が進化先)
```

## コンポーネント

| # | もの | 実装 | 備考 |
|---|---|---|---|
| 1 | アップロードAPI | API GW + Lambda | S3 presignedマルチパートURL発行。GB級・中断再開OK |
| 2 | 動画バケット | S3 `videos/` | ライフサイクル管理（下記） |
| 3 | 抽出 | Lambda + ffmpegレイヤ | 2fps・縮小でデコード、フレーム間差分で「1.5秒以上静止＆前回採用と差分あり」を採用。手が写っている間は差分が大きく自然に除外。1時間動画で数分・$0.02〜0.05 |
| 4 | 認識ループ | Lambda | 既存 `/recognize` を並列数制限(要API側と合意)つきで呼ぶ。`waku_hint`が実装されれば1フレーム目の枠を使い回す |
| 5 | 手順復元 | Lambda (Node.js) | `public/js/renzoku.js` は依存ゼロで書いてあるのでそのまま import できる。二歩等の警告・読み飛ばしブリッジ・駒数保存則トラッキング込み |
| 6 | ジョブ状態 | DynamoDB + API GW | `{status, progress, result_url}`。フロントはjob_idをlocalStorageに保存し「閉じても後で見られる」 |
| 7 | 補正エディタ | フロント | renzoku.htmlの結果画面を拡張（N手目修正→以降を再復元） |

## ストレージ方針: 動画は消さずに階層を落とす

即削除はしない。理由:
- **再処理資産**: 認識モデルや復元ロジックが改善したら、過去の動画から棋譜を作り直せる（再アップロード不要）。これはサービス改善のたびに全ユーザーの棋譜が良くなることを意味する
- **ユーザー価値**: 「あの場面を見返したい」に応えられる
- **コストは階層化すれば誤差**（下記試算）

ライフサイクルルール（S3で自動化・コード不要）:

| 期間 | 階層 | 単価(東京・概算) | 用途 |
|---|---|---|---|
| 0〜30日 | S3 Standard | $0.025/GB·月 | 処理・再処理・見返しのホット期間 |
| 30〜90日 | Standard-IA | $0.0138/GB·月 | たまの見返し（取り出し$0.01/GB） |
| 90日〜 | Glacier Instant Retrieval | $0.005/GB·月 | 長期保管。ミリ秒で取り出せる（$0.03/GB）ので「見返し」体験は変わらない |

- キーフレーム(20MB/局)と結果JSONは小さいのでStandardに置きっぱなしでよい
- 削除はユーザーリクエスト起点（マイページ or 問い合わせ）+ B2B契約では「大会終了後N日で削除」等を選べるようにする

**試算**: 1局=1GB、月30局を1年間continued運用した定常状態
≈ Standard 30GB($0.75) + IA 60GB($0.83) + Glacier IR 270GB($1.35) ≈ **月$3程度**。
月100局でも$10前後。動画保持はコスト問題にならない。

## プライバシー（行橋ガイドライン整合）

対局動画には対局者・周囲の人が写り得る。行橋支部の撮影ガイドライン
（映り込み最小化・SNS投稿時の配慮）と整合させるため:

1. アップロード時に「動画は棋譜復元と品質改善のために保存されます。削除依頼はいつでも可能」を明示・同意取得
2. 復元結果として**外に出るのはキーフレーム由来の盤面情報のみ**（動画そのものは公開されない）
3. B2B（大会運営）向けには「処理後即削除」オプションを契約で選べるようにする
4. 学習データとしての利用（API提供側が欲しいはず）は**別途の明示同意**を設けてから。無断利用しない

## API提供側との関係（更新後）

- `POST /recognize`（単発・既存）を使い続ける。**バッチAPI(旧R1)・ジョブAPI(旧R2)の要求は撤回**（自前化）
- 残る要求は `API_REQUIREMENTS.md` 参照（確信度・レート合意・waku_hint・単価）

## 実装順

1. **アップロード+抽出**（#1〜#3）: 動画→キーフレームがS3に並ぶところまで。抽出品質（取りこぼし/重複率）を実対局動画で検証
2. **認識ループ+手順復元**（#4〜#6）: renzoku.jsの移植。ジョブAPIまで通してフロント接続
3. **補正エディタ強化**（#7）: N手目修正→以降再復元、確信度ハイライト（API側R3待ち）
